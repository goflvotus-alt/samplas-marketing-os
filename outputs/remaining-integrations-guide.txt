다음 1: Meta Ads 연결
필요 값
- META_AD_ACCOUNT_ID=act_숫자
- META_ACCESS_TOKEN에 ads_read 권한 포함

어디서 확인
- Meta Business Settings
- 광고 계정
- 광고 계정 ID 복사
- .env의 META_AD_ACCOUNT_ID에 입력

연결 테스트
- 서버 실행 후 열기:
  http://127.0.0.1:8787/api/meta-ads/summary

주의
- 광고 계정 ID는 반드시 act_로 시작하는 형식으로 넣습니다.
- ads_read 권한이 없으면 Meta Ads API는 권한 오류가 납니다.

다음 2: Cafe24 주문 데이터 연결
필요 값
- CAFE24_MALL_ID
- CAFE24_CLIENT_ID
- CAFE24_CLIENT_SECRET
- CAFE24_ACCESS_TOKEN
- CAFE24_REFRESH_TOKEN

어디서 발급
- Cafe24 Developers
- Apps
- Admin API 앱 생성
- 주문 조회 권한 활성화
- OAuth 인증으로 access token 발급

연결 테스트
- 서버 실행 후 열기:
  http://127.0.0.1:8787/api/cafe24/orders

자동 토큰 갱신
- 주문 API 호출 중 access token이 만료되면 서버가 refresh token으로 자동 갱신을 시도합니다.
- 수동 갱신 테스트:
  http://127.0.0.1:8787/api/cafe24/refresh-token
- 갱신 성공 시 .env의 CAFE24_ACCESS_TOKEN이 자동으로 교체됩니다.
- refresh token까지 만료되면 Cafe24 OAuth 재인증이 필요합니다.

Cafe24 OAuth 재인증
- Cafe24 Developers 앱 설정의 Redirect URI에 아래 주소를 추가하세요.
  http://127.0.0.1:8787/api/cafe24/oauth/callback
- 서버를 켠 상태에서 아래 주소를 여세요.
  http://127.0.0.1:8787/api/cafe24/oauth/start
- Cafe24 권한 승인 후 callback으로 돌아오면 새 access token과 refresh token이 .env에 자동 저장됩니다.
- 그 다음 주문 API를 다시 확인하세요.
  http://127.0.0.1:8787/api/cafe24/orders

권장 운영 구조: Render를 Cafe24 토큰 주인으로 사용
- Render와 로컬이 같은 Cafe24 refresh token을 각각 갱신하면 한쪽 토큰이 무효화될 수 있습니다.
- 그래서 Cafe24 토큰은 Render 한 곳에서만 갱신하고, 로컬은 Render API를 프록시로 읽는 구조를 권장합니다.
- 로컬 .env에 아래 값을 넣으면 로컬은 Cafe24 토큰을 직접 쓰지 않습니다.
  CAFE24_PROXY_BASE_URL=https://samplas-meta-dashboard.onrender.com
  CAFE24_PROXY_ORDERS_PATH=/api/cafe24/orders
- 이 경우 로컬 /api/cafe24/orders는 Render의 /api/cafe24/orders를 호출합니다.
- Render 쪽에 보호 토큰을 걸었다면 로컬 .env에 CAFE24_PROXY_SECRET도 같은 값으로 넣으세요.
- 현재 Render 앱이 Basic Auth로 보호되어 있으면 로컬 .env에 아래도 넣으세요.
  CAFE24_PROXY_BASIC_AUTH=아이디:비밀번호

운영 원칙
- Meta 광고 매출은 참고 지표로만 사용합니다.
- 실제 매출 판단은 Cafe24 주문 API 기준으로 봅니다.
- Instagram에서 제공하지 않는 지표는 수집 불가 또는 수동 입력 필요로 표시합니다.
